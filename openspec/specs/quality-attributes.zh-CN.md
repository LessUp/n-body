# 能力：质量属性

## 概述

定义 N 体模拟系统的非功能性需求，包括性能、可靠性和可用性。

---

## 新增需求

### 需求：帧率性能

系统应满足各种粒子数量的指定帧率目标。

#### 场景：10 万粒子性能
- **当** 模拟 10 万粒子时
- **则** 帧率应至少为 60 FPS

#### 场景：100 万粒子 Barnes-Hut 性能
- **当** 使用 Barnes-Hut 算法模拟 100 万粒子时
- **则** 帧率应至少为 25 FPS

#### 场景：100 万粒子空间哈希性能
- **当** 使用空间哈希算法模拟 100 万粒子时
- **则** 帧率应至少为 60 FPS

---

### 需求：内存效率

系统应保持高效的内存使用。

#### 场景：每粒子内存
- **当** 模拟 100 万粒子时
- **则** 总内存使用应小于 200 MB

#### 场景：内存布局
- **当** 分配粒子数据时
- **则** 应使用结构数组布局以实现 GPU 合并访问

---

### 需求：能量守恒精度

系统应在长时间模拟中保持能量守恒。

#### 场景：能量漂移限制
- **当** 模拟运行 1000 步时
- **则** 能量漂移应小于 1%

#### 场景：辛积分
- **当** 使用 Velocity Verlet 积分器时
- **则** 相空间体积应被保持

---

### 需求：测试覆盖率

系统应保持全面的测试覆盖率。

#### 场景：代码覆盖率
- **当** 执行测试时
- **则** 核心组件覆盖率应超过 90%

#### 场景：基于属性的测试
- **当** 运行基于属性的测试时
- **则** 所有正确性属性应通过

---

### 需求：可靠性属性

系统应满足定义的正确性属性。

#### 场景：力计算正确性
- **当** 计算力时
- **则** 大小应近似 G·m₁·m₂/(r² + ε²)

#### 场景：Barnes-Hut 树结构
- **当** 构建八叉树时
- **则** 树应包含恰好 N 个粒子并保持质量守恒

#### 场景：力方法等价性
- **当** Barnes-Hut 使用低 θ 时
- **则** 结果应收敘到直接 N²

#### 场景：CUDA-GL 数据完整性
- **当** 数据在 CUDA 和 OpenGL 之间传输时
- **则** 数值应保持不变无损坏

---

### 需求：构建简单性

系统应提供简单、规范的构建路径，并同时支持验证自动化与编辑器/LSP 集成。

#### 场景：规范构建路径
- **当** 贡献者遵循主构建说明时
- **则** 仓库应提供一条清晰文档化的基于 CMake 的构建路径，并在文档、脚本和自动化之间保持一致

#### 场景：依赖版本受控
- **当** 构建依赖拉取的工具、库或 Actions 时
- **则** 其版本应被固定或显式约束，并以可文档化、可复现的方式维护

#### 场景：LSP 来源于主构建
- **当** 为 C++ 或 CUDA 开发配置编辑器工具时
- **则** 推荐配置应使用主构建路径生成的产物，包括 `compile_commands.json` 或等价编译数据库

---

### 需求：API 文档

系统应提供完整的 API 文档。

#### 场景：Doxygen 覆盖
- **当** 运行 Doxygen 时
- **则** 所有公共 API 应被文档化

#### 场景：使用示例
- **当** 开发者需要指导时
- **则** 应至少有 4 个示例程序可用

---

### 需求：文档完整性

所有面向读者和贡献者的文档都应准确、无重复，并且经过针对本项目的有意策划。

#### 场景：规范文档保持一致
- **当** 仓库文档描述架构、工作流、安装或示例时
- **则** 每类用途都应只有一个规范版本，陈旧重复内容应被删除或缩减为指针

#### 场景：双语核心材料保持同步
- **当** 更新核心规格或主要入门文档时
- **则** 其必需的双语对应文档也应同时更新以保持同步

#### 场景：已发布链接保持有效
- **当** 文档引用仓库内文件、Pages 路由或 GitHub 表面时
- **则** 这些链接应解析到当前有效的目标

---

### 需求：验证信号质量

仓库自动化应强调与真实支持工作流一致的高信号验证。

#### 场景：CI 检查有意义
- **当** 工作流在持续集成中运行时
- **则** 每个必需作业都应验证一个受支持的仓库行为，而不是执行纯装饰性或冗余检查

#### 场景：Pages 构建范围受控
- **当** 一个改动未触及站点或已发布文档表面时
- **则** Pages 工作流应保持空闲

---

### 需求：仓库可维护性

仓库应通过减少配置漂移、重复指导和不必要的工具复杂度来最小化维护开销。

#### 场景：配置来源保持一致
- **当** 格式化、工作流或开发说明在多个位置定义时
- **则** 仓库应保持这些来源一致，或将其合并为更小的一组规范入口

#### 场景：可选工具保持轻量
- **当** 考虑引入插件、MCP 集成或辅助工具时
- **则** 只有在其能明显改善项目工作且不会带来过高维护或上下文成本时才应采用
