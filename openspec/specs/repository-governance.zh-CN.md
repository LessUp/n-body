# 能力：仓库治理

## 概述

定义 OpenSpec 驱动开发、文档质量、AI 指令文件、自动化卫生、GitHub 对外呈现以及项目收尾工作流的仓库级运行模型。

---

## 新增需求

### 需求：OpenSpec 单一事实来源

仓库应将 `openspec/` 作为新开发、维护和收尾工作的唯一活跃规格系统。

#### 场景：活跃规格无歧义
- **当** 贡献者或 AI 代理查找当前需求时
- **则** 规范的权威来源应为 `openspec/specs/`，任何旧规格树都应标记为仅供历史参考

#### 场景：变更工作从 OpenSpec 开始
- **当** 一个功能、修复、重构或收尾任务会改变仓库行为或工作流时
- **则** 在开始实现之前，该工作应从 OpenSpec 提案或现有活跃 OpenSpec 变更开始

---

### 需求：仓库结构治理

仓库应保持经过设计的目录结构，使文档、站点内容、变更日志内容和工程配置各司其职，而不会漂移成重复镜像。

#### 场景：重复内容被清理
- **当** 两个文件或目录服务于同一受众和用途时
- **则** 应保留一个规范入口，另一个应被删除、重定向或缩减为指针

#### 场景：历史材料被清晰隔离
- **当** 保留旧的目录重构总结、遗留规格或已被替代的文档时
- **则** 它们应被标注为历史或遗留内容，并且不得表现为当前工作流

---

### 需求：AI 协作指令

仓库应提供项目定制化的 AI 指令文件，并确保这些文件在工作流、仓库结构和工程预期上保持一致。

#### 场景：指令文件保持一致
- **当** `AGENTS.md`、`CLAUDE.md` 或 Copilot 指令文件描述开发流程时
- **则** 它们应全部指向同一套 OpenSpec 优先工作流，并且不得引用互相矛盾的仓库结构

#### 场景：拒绝通用模板
- **当** 新增或更新 AI 指令文件时
- **则** 其内容应提供真正有助于本仓库工作的项目特定指导，而不是通用助手模板

---

### 需求：工作流自动化质量

仓库应仅保留有意义的自动化，并且应将工作流触发范围限制在真正值得执行的改动上。

#### 场景：工作流验证真实路径
- **当** CI 或 Pages 工作流执行时
- **则** 它们应验证文档化的构建、测试、文档或站点流程，而不是仅依赖仪式化的字符串检查

#### 场景：工作流噪音最小化
- **当** 仓库变更没有触及某个工作流所负责的表面时
- **则** 该工作流不得触发

---

### 需求：开发环境基线

仓库应围绕规范构建路径和可复用的语言工具，提供精简的开发工具基线并进行文档化。

#### 场景：LSP 配置跟随构建系统
- **当** 贡献者配置编辑器辅助能力时
- **则** 推荐的 C++/CUDA LSP 路径应来源于主 CMake 构建生成的编译数据库

#### 场景：本地护栏可用
- **当** 贡献者准备本地提交变更时
- **则** 仓库应提供轻量级 hooks 或等价自动化以执行最高价值的质量检查

---

### 需求：GitHub 项目呈现

仓库应在 GitHub 和 GitHub Pages 上清晰呈现自己，使新用户能够快速理解项目是什么、为什么重要以及下一步该去哪里。

#### 场景：仓库元数据经过策划
- **当** 用户查看仓库 About 区域时
- **则** 描述、主题标签和主页链接应准确反映项目，并在可用时包含发布后的 Pages URL

#### 场景：Pages 承担展示职能
- **当** 用户访问 GitHub Pages 时
- **则** 落地页应解释项目价值、突出技术差异点，并将用户引导到文档、示例和源码

---

### 需求：收尾工作流模型

仓库应文档化一套轻量的收尾工作流，在不引入不必要分支或工具负担的前提下支持最终稳定化工作。

#### 场景：优先使用长时会话实现
- **当** 为大范围清理或收尾任务规划 AI 辅助实现时
- **则** 文档化建议应优先长时 `autopilot` 执行，而不是常规使用 `/fleet`，除非任务确实高度适合并行

#### 场景：审查关口明确
- **当** 一个重大的结构性或治理性重构准备合并时
- **则** 工作流应要求显式审查步骤，例如在最终完成前执行 `/review`
