## MODIFIED Requirements

### Requirement: Build Simplicity
The system SHALL provide a simple, canonical build path that also supports validation automation and editor/LSP integration.

#### Scenario: Canonical build path
- **WHEN** a contributor follows the primary build instructions
- **THEN** the repository SHALL provide one clearly documented CMake-based build path that remains consistent across docs, scripts, and automation

#### Scenario: Dependency versions are controlled
- **WHEN** the build depends on fetched tools, libraries, or actions
- **THEN** their versions SHALL be pinned or explicitly bounded in a way that is documented and reproducible

#### Scenario: LSP derives from the primary build
- **WHEN** editor tooling is configured for C++ or CUDA development
- **THEN** the recommended setup SHALL use artifacts generated by the canonical build path, including `compile_commands.json` or an equivalent compile database

### Requirement: Documentation Completeness
All reader-facing and contributor-facing documentation SHALL be accurate, non-duplicated, and intentionally curated for this project.

#### Scenario: Canonical documents stay consistent
- **WHEN** repository documentation describes architecture, workflow, setup, or examples
- **THEN** there SHALL be one canonical version for each purpose and stale duplicates SHALL be removed or reduced to pointers

#### Scenario: Bilingual core materials stay in parity
- **WHEN** a core specification or primary onboarding document is updated
- **THEN** its required bilingual counterpart SHALL be updated to preserve parity

#### Scenario: Published links remain valid
- **WHEN** documentation references internal files, Pages routes, or GitHub surfaces
- **THEN** those links SHALL resolve to valid current targets

## ADDED Requirements

### Requirement: Validation Signal Quality
Repository automation SHALL emphasize high-signal validation that matches the real supported workflow.

#### Scenario: CI checks are meaningful
- **WHEN** a workflow runs in continuous integration
- **THEN** each required job SHALL validate a supported repository behavior rather than perform purely cosmetic or redundant checks

#### Scenario: Pages builds stay scoped
- **WHEN** a change does not affect the site or published documentation surface
- **THEN** the Pages workflow SHALL remain idle

### Requirement: Repository Maintainability
The repository SHALL minimize maintenance overhead by reducing configuration drift, duplicated guidance, and unnecessary tool complexity.

#### Scenario: Configuration sources agree
- **WHEN** formatting, workflow, or development instructions are defined in multiple places
- **THEN** the repository SHALL keep those sources aligned or consolidate them into a smaller canonical set

#### Scenario: Optional tooling remains lightweight
- **WHEN** a plugin, MCP integration, or auxiliary tool is considered
- **THEN** it SHALL only be adopted if it clearly improves project work without imposing disproportionate maintenance or context cost
